# The four phases (P0–P3)

The narrate-and-point detail behind the [index](README.md). The canonical *intentions* of each phase
live in the coach's
[`engagement-model.md`](../../skills/semantic-consulting-coach/references/engagement-model.md);
this file maps each phase to its owning skill(s) and the boundary that protects it. It restates no
skill's rules.

The free→paid boundary is the spine of the model: **orientation is free** ("is this relevant?");
**deciding is paid** ("what do we do, in what order?"). Answering the second for free leaks
intellectual capital.

---

## P0 — Orientation (free, bounded)

**Question:** "Is this even relevant to us?" · **Commercial:** unpaid, deliberately shallow.

Mutual recognition — are we relevant to each other? Teach generously but never deeply customise; the
moment talk shifts from *"is this relevant?"* to *"what should we do?"*, the boundary is reached and
the next step is a **proposal**, not more free discussion.

- **Owner:** [`semantic-consulting-coach`](../../skills/semantic-consulting-coach/SKILL.md)
  — it holds the free→paid line and coaches *whether/how* to run P1.

---

## P1 — Decision (paid, fixed frame)

**Question:** "What do we do first, and why?" · **Commercial:** paid, fixed frame.

The keystone. A short paid engagement that moves an organisation from interest to a **safe, informed
decision about what to build next** — before any pilot or programme. Its value is
**decision-readiness**, and its deliverable is the **Decision Package**.

The skills for P1, and what each owns:

- [`proposal-writing`](../../skills/proposal-writing/SKILL.md) — the **entry skill**:
  qualifies the need, frames the offer, prices it as a fixed frame, and writes the **proposal + SoW**
  with an explicit in/out scope boundary.
- [`estimation`](../../skills/estimation/SKILL.md) — the **pricing/scoping** discipline
  (PERT + work breakdown + ranges + contingency) that `proposal-writing` invokes for the numbers;
  independently triggerable.
- [`decision-package`](../../skills/decision-package/SKILL.md) — **produces** the
  Decision Package: recommendation, in/out scope, pilot→scale roadmap, buy/build/defer, and the
  ready-to-contract execution brief that hands off to P2.

**Boundary:** execution is *always* a separate engagement. The Decision Package earns the right to
say *"from here, we can execute, or you can take this and execute with someone else."*

---

## P2 — Execution (paid, scoped)

**Question:** "How do we build this well?" · **Commercial:** paid, separately scoped.

Build what P1 decided. Valid only once P1 is complete. This is the **build tier** — the two-tier
methodology takes over here.

- **Pointer:** [`../ai-coding/two-tier-methodology.md`](../ai-coding/two-tier-methodology.md) (the
  PROJECT and EPIC tiers). The P1 → P2 hand-off is the Decision Package's execution brief, which
  becomes the parent of the first architecture/requirement artefacts (the golden thread).

---

## P3 — Partnership & Evolution (paid, optional)

**Question:** "How do we keep this alive?" · **Commercial:** paid, optional.

Sustain, scale, and evolve what was built: governance support, semantic operations, capability
building. Post-decision continuity.

- **Owner:** *to be shaped* — no skill yet. Governance/ops practices will be crystallised after the
  dogfood engagement.
